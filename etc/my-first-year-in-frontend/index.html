프론트엔드 개발 1년 차의 회고

Etc2024-11-23 · 9 min read

들어가며

작년 11월에 입사하고 일 년이라는 시간이 지나서 회고를 작성하려 합니다. 프론트엔드 개발자로 일하면서 어떠한 일이 있었는지 정리하고 그 과정에서 얻은 점과 아쉬운 점을 정리했습니다.

1. 첫 입사! 어떤 걸 할 수 있을까? (2023.11 ~ 2023.12)

레모네이드 CIC 와 스노우볼 CIC 업무에 참여했다. 서비스로는 가벼운학습지 와 제로베이스 를 운영했다. 두 프로젝트 모두 Next.js(Pages Router) 와 TypeScript 를 기반으로 한 프로젝트였습니다.

첫 PR 은 프로젝트를 로컬 환경에 세팅하면서 README 파일을 수정한 것이었습니다. 이 과정에서 새로운 로컬 세팅 가이드라인 문서도 작성했습니다. 이후에는 간단한 이슈부터 시작하여 홈페이지 하단의 이메일 텍스트 변경 등의 작업을 진행했습니다. PR 생성 시 assignees 를 직접 설정하는 것이 어색하여 자꾸 놓치게 되어 자동화 workflow 를 추가했습니다.

운영 이슈에 참여하면서 백오피스의 사용법을 익혀나갔습니다. 연말이 성수기에는 마케팅 급건 관련 작업도 진행했습니다. 당일 요구사항 파악 후 즉시 변경하여 배포하는 핫픽스도 경험했습니다. 당시에는 작은 수정사항이어도 바로 배포하는 것 자체가 긴장되었습니다. 지금 생각해보면 이때의 경험으로 코드의 사이드 이펙트를 더 꼼꼼히 검토해보게 되어 좋은 경험이었다고 생각합니다.

2주에 한 번씩 정기 배포를 진행하는 프로세스가 있어서 이를 위한 배포 준비도 했습니다. 첫 배포를 준비할 때는 뭔가 놓치는 부분이 있을까 걱정되어 배포 프로세스 문서를 정독하며 차근차근 준비했습니다.

2. 수습 3개월 끝, 다른 서비스 체험 (2024.01 ~ 2024.02)

클린업 작업에 참여하여 Query Key Factory 를 도입해서 Query Key 구조를 개선했습니다. 이 과정에서 컨벤션이 변경되어 문서를 최대한 상세하게 작성하려 노력했습니다.

2월에는 패스트캠퍼스 의 LMS 관련하여 Vue 기반 프로젝트를 Next.js 기반으로 이관하는 작업을 참여했습니다. Vue 프로젝트를 처음 접해서 새로운 문법이 어색했지만 그래도 무사히 작업을 마무리했습니다.

React Query 는 이를 캐시하고, 결과 페이지에서는 캐시한 데이터를 바라봐서 발생한 오류였다.

기존 Recoil 로 작성한 일부 로직을 Zustand 로 전환했습니다.

백오피스 업무에도 참여하면서 조금씩 백오피스 사용법을 익혀나갔습니다.

팀 내에서 Next.js Using App Router 문서 읽기 스터디에도 참여했습니다.

3. 운영 이슈와 클린업 (2024.03 ~ 2024.05)

이후에는 다시 가벼운학습지와 제로베이스를 운영했습니다.

새로운 클린업 작업에 참여해서 Playwright 로 e2e 테스트를 작성했습니다. 다만 테스트 코드 작성 직후 조직 개편으로 인해 실제 운영 과정에서 테스트 코드를 더 발전시키지 못한 점은 아쉬움으로 남아있습니다.

운영 이슈를 진행하며 GNB 카테고리 메뉴를 기존 2depth 에서 3depth 까지 확장하는 작업에 참여했습니다.

사용자 드래그하여 가로 스크롤이 가능하도록 공통 hook 을 추가했습니다.

팀 내에서 MDN HTTP 문서 읽기 스터디에 참여했습니다.

4. 조직 개편 및 새로운 서비스 시작 (2024.06 ~ 2024.07)

조직 개편으로 인해 6월부터는 패스트캠퍼스에 합류했습니다. 패스트캠퍼스는 Next.js(App Router) 와 TypeScript 를 사용하고 있습니다.

이전에는 비교적 작은 단위의 이슈가 많았는데 패스트캠퍼스에서는 여러 달에 걸친 에픽 단위의 이슈를 진행했습니다. 6월부터 현재까지 크게 세 개의 에픽을 진행했습니다.

  • BX/UX 개편(메인 페이지에 실시간 BEST 인기 강의와 같은 여러 자동화 섹션 생성 및 찜하기 기능 도입)
  • 상세 페이지 개편(상세 페이지 자동화 템플릿 도입)
  • 질의응답 게시판 고도화(커뮤니티 도입)

6월에는 BX/UX 개편 작업에 참여했습니다. 개인적으로 이 기간이 에픽 단위의 이슈가 업무량이 많은데 새로운 서비스의 도메인 지식을 익히는 것까지 함께 해야 해고 팀이 바뀌어서 새로운 팀에 적응도 해야 해서 올 한 해 중에서 가장 혼란스럽고 어렵고 힘들었습니다.

에픽 단위의 이슈를 진행하며 운영 이슈에 대응해서 새로운 서비스의 도메인 지식을 익혀나가고 있습니다.

사내에서 사용하는 패키지의 개선 작업에 참여했습니다.

스켈레톤 UI를 도입했습니다.

5. 신규 기능 도입 (2024.08 ~ 2024.10)

상세 페이지 자동화 템플릿 도입 작업을 진행했습니다.

백오피스에서는 사용자가 엑셀 파일을 업로드하고 UI 상에서 내용을 확인할 수 있는 기능을 추가했습니다.

이 과정에서 데이터 검증 오류를 상세하게 작성하여 사용자가 어떤 데이터에서 오류가 발생하는지 쉽게 파악할 수 있도록 했습니다.

그리고 실제 서비스에서는 이 데이터를 아코디언 UI 형태로 커리큘럼 데이터를 노출하는 작업을 진행했습니다.

사내 패키지도 수정하면서 ejs를 처음 접하게 되었습니다.

영상 미리보기 기능을 도입하면서 관련 도메인 지식도 익힐 수 있었고, 현재는 디자인시스템 도입을 진행하고 있습니다

에픽 단위의 새로운 기능을 개발할 때는 기획서와 피그마를 꼼꼼히 검토하여 놓치는 부분이 없도록 신경 썼습니다.

백오피스 작업을 하면서 Vue를 더 깊이 이해하게 되었고, 리팩토링 작업을 통해 반복되는 공통 로직을 hook으로 분리했습니다.

회고를 작성하며 확인해보니 총 303개의 PR과 2,534개의 커밋을 작성했습니다. 문득 일 년에 보통 몇 개의 PR을 생성하는지 궁금해졌습니다.

열심히 한 걸까? 갑자기 문득 일 년에 몇 개의 PR을 생성하는지 궁금해졌다.

이 과정에서 겪은 걸 KPT 회고로 정리해보려 한다.

6. 커뮤니티 오픈 (2024.11 ~ 2024.12)

커뮤니티가 여태 진행한 프로젝트 중 가장 일정이 촉박했다. 사실상 불가능한 일정이라고 생각했는데 무사히 배포까지 마쳐서 다행이다. 이제 QA 및 안정화 작업과 2차, 3차 배포를 앞두고 있다.

팀이 다 함께 주말에 근무를 하기도 할 정도로 업무량은 많았지만 그래도 개인적으로 제일 재미있었다.

재미있었던 이유는 공통 로직과 관련된 업무를 맡아서 즐거웠다. 퍼널 UI 관련 hook 및 컴포넌트 만들기, 무한 스크롤 공통 hook 만들기, 목록 UI 공통 컴포넌트 만들기 등이 있었다. 이렇게 여러 케이스를 고려하는 게 좋고 재미있다. 다른 개발자의 사용 경험도 고려하는 게 좋다. 잘못 사용했을 경우를 대비해서 타입 에러를 발생시키는 등 패키지와 같은 곳에서는 왜 타입스크립트를 잘 사용해야 하는지를 알게된 것도 좋았다. 재미있으니 자발적으로 집에 와서도 더 고도화를 하거나 발전시키고 싶다는 생각이 계속 들었다. 하기 싫을 일을 해야 해서 하는 게 아니라 정말 일이 재미있어서 하는 느낌이었다.

나는 그동안 프론트엔드의 여러 분야 중 내가 어떤 일을 하고 싶은지 잘 몰랐다. 드디어 내가 재미있어하고 좋아하고 잘할 수 있는 분야를 알게된 거 자체가 큰 수확이라고 생각한다. 물론 하기 싫은 일도 열심히 하겠지만 그래도 내가 더 흥미가 있고 좋아하는 일을 더 열심히 하게 되는 건 어쩔 수 없지 않나? 이제 날 소개하는 문구를 어떻게 정해볼 수 있을지 좀 생각이 든다.

  • 저는 다른 개발자의 경험도 생각하는 걸 좋아해요.
  • 여러 케이스를 고려하는 걸 좋아해요.
    • 다양한 사용 케이스 고려 필요
    • 추상화 능력 중요
    • 재사용성과 확장성 고려
    • 사용자(다른 개발자들) 경험 중시

1. Keep

그래서 어떤 일 년이었는지?

돌이켜 생각해보니 많은 일이 있었던 정신없이 바쁜 일 년이었던 거 같다.

많은 분들에게 큰 도움을 받았는데 첫 일 년 동안 느낀 걸 잊지 않고 기억했다가 나중에 내가 리더가 된다면 이를 다 돌려줄 수 있으면 좋겠다.

그래도 일 년 동안 전보다 어떤 점에서 발전했는지 그리고 앞으로도 계속 가져가면 좋을 자세는 어떤 점이 있는지를 정리해보려 한다.

1-1. 코드 리뷰

조금 성장했다고 느낀 건 코드 리뷰이다. 입사 초에는 코드 리뷰에 적극적으로 참여하고 싶어서 꼼꼼하게 코드를 검토해봐도 리뷰할 게 없었다. 그래도 이젠 조금씩 코드 리뷰를 통해 의견을 남기고 있다. 조금이나마 코드가 개선되는 것을 보면 처음 입사했을 때 보단 나아진 거 같다. 이렇게 토의를 통해 더 나은 코드 방향에 대해 논의하는 과정이 좋고 재미있다. 개발자에게 코드 리뷰는 좋은 문화라고 생각한다. 코드 리뷰 멘토링에 참여하고 싶은 마음도 있다.

1-2. 생각 정리

  • 사실 번아웃에 공감이 가지 않았다. 이젠 미리 관리하는 게 중요하다는 것을 느꼈다. 내가 무너지지 않도록
  • 이전에는 무조건 무너지지 않는 것만 중요하다고 생각했는데, 이젠 무너져도 다시 일어나는 게 중요하다고 생각한다.
  • 일반 서적을 조금 읽었다. 많이는 못 읽었지만 읽으면서 혼자 끄적이는 게 도움이 된다.
  • 엄청난 걸 적는 게 아니라 혼자 생각하는 걸 정리한다. 그러면서 미래에 대해 계획도 조금씩 추가한다.
  • 이러면서 스스로 정리되는 게 참 좋다. 원래 대학 시절부터 철학에 대한 관심이 조금 있었는데 글을 읽으면서 쓰기도 한다는 게 좋다. 스스로 정리가 되면서 정신적으로 정리가 된다.
  • 근데 또 혼자 불안감을 느끼며 자괴감을 느낀다.
  • 요즘 생각을 비우기 위해 책을 읽는다. 책을 읽는 동안은 잡생각이 사라지고 안정감을 느낀다.
  • 스스로를 안정된 상태로 만드는 방법은 명확하게 알게된 거 같아서 좋다. 이 방법은 계속 사용하고 싶다.

2. Problem

스스로에게 아쉬웠던 점을 정리해봤다. 내년에 회고를 작성할 땐 이러한 사항이 반복되지 않았으면 좋겠다.

  • 한 해 동안 기술 역량을 더 끌어올리지 못한 게 아쉽다. 더 많은 개인 시간을 투자하여 학습하지 못한 것이 아쉽다.
  • 아직도 나만의 개발 학습 방법을 정립하지 못한 것 같다. 내년 한 해 동안 오랜 기간 나에게 힘이 되어줄 학습 방법을 정립하고 싶다.
  • 개발할 때 사소한 선택에도 주관을 가지려 노력한다. 근데 어렵다..

4. 단순 Thinking

  • 확실히 도메인 지식은 많이 쌓은 거 같다. 백오피스를 전보다는 익숙하게 사용한다.

  • 그래도 아직 모르는 도메인 지식이 많다. 운영 이슈를 하면서 하나씩 배워가고 있다.

  • 그리고 내가 팀 내의 커뮤니케이션을 이어주는 역할을 하다는 평을 들어서 좋았다.

  • 구두 커뮤니케이션과 텍스트 커뮤니케이션 그 사이를 이어주는 역할을 하고 있다.

  • 필요한 정도의 팔로업을 잘하는 거 같다고 하셨다.

  • 일 년 동안 잘 지낸 게 맞을까? 난 어떻게 지낸걸까?

  • 앞으로 오랜 기간 개발자를 하면서 살텐데 커리어를 잘 쌓아나갈 수 있을까?

  • 아직 많은 게 서툴다 직장 생활도 인간 관계도 모든 것이 다.

  • 최근에 공통 로직과 관련된 업무를 했는데 너무 재미있었다.

  • 퍼널 UI 관련 hook 및 컴포넌트 만들기, 무한 스크롤 공통 hook 만들기, 목록 UI 공통 컴포넌트 만들기

  • 이렇게 여러 케이스를 고려하는 게 좋고 재미있다. 다른 개발자의 사용 경험도 고려하는 게 좋다.

  • 잘못 사용했을 경우를 대비해서 타입 에러를 발생시키는 등 패키지와 같은 곳에서는 왜 타입스크립트를 잘 사용해야 하는지를 알게된 것도 좋았다.

  • 그리고 실제로 잘한다는 평도 받아서 더 좋았다. 근데 그래서 좋은 건 아니고 원래도 재미있었다. 흥미가 있었다.

  • 재미있고 집에 와서도 더 고도화를 하거나 발전시키고 싶다는 생각이 계속 들었다.

  • 하기 싫을 일을 해야 해서 하는 게 아니라 정말 일이 재미있어서 하는 느낌이었다.

  • 이러한 일을 최대한 많이 할 수 있는 환경에 있으면 좋을 거 같다.

  • 나는 그동안 프론트엔드의 여러 분야 중 내가 어떤 일을 하고 싶은지 잘 몰랐다.

  • 드디어 내가 재미있어하고 좋아하고 잘할 수 있는 분야를 알게된 거 자체가 큰 수확이라고 생각한다.

  • 물론 하기 싫은 일도 열심히 하겠지만 그래도 내가 더 흥미가 있고 좋아하는 일을 더 열심히 하게 되는 건 어쩔 수 없지 않나?

  • 이제 날 소개하는 문구를 어떻게 정해볼 수 있을지 좀 생각이 든다.

  • 저는 다른 개발자의 경험도 생각하는 걸 좋아해요.

  • 여러 케이스를 고려하는 걸 좋아해요.

    • 다양한 사용 케이스 고려 필요
    • 추상화 능력 중요
    • 재사용성과 확장성 고려
    • 사용자(다른 개발자들) 경험 중시

  • 주위 사람만 봐도 다 본받고 싶은 점이 다르다. 내가 어떤 동료, 개발자가 되고 싶은지를 생각하고 나아가는 게 좋은 거 같다. 정답은 없다 내 생각은 뭘까? 나는 결국 어떤 사람이 되고 싶은 걸까!

  • 나는 나중에 팀을 이끌어 나갈 수 있는 좋은 리더가 되고 싶다. 어떤 걸 학습하는 게 좋을까? 일단 기술 역량을 높이는 건 나중에 리더가 아닌 어떤 방향으로 나아가도 필요할 거 같다.

  • 언젠가 세미나에서 발표도 하고 싶다. 아직은 발표하고 싶은 주제가 없다.

  • 취업을 하면 취미에 대해 관심을 가지는 후기 글을 많이 봤을 때 신기했다.

5. Goal

앞으로 일 년 후에 가장 이루고 싶은 목표만 작성했다.

3. Try

아쉬웠던 점을 개선하기 위해서는 어떤 해결책이 있을지 그리고 당장 실행할 수 있는 것을 정리해봤다.

아쉬웠던 점을 개선하기 위해서는 어떤 해결책이 있을지 그리고 당장 실행할 수 있는 것을 정리해봤다.

  • 기술 역량 향상을 위한 구체적인 계획

    • 매주 1-2개의 기술 블로그 글 작성하기
    • 공통 컴포넌트/훅 개발 경험을 바탕으로 오픈소스 프로젝트 참여하기
    • 매월 한 가지 새로운 기술 스택 깊이 있게 학습하기

  • 효과적인 학습 방법 정립

    • 업무 중 마주치는 문제들을 노션이나 블로그에 체계적으로 정리하기
    • 주간 회고를 통해 배운 점과 개선할 점 돌아보기
    • 동료 개발자들과 스터디 그룹 만들어 정기적으로 지식 공유하기

  • 개발 결정에 대한 주관 기르기

    • 코드 리뷰시 “왜 이렇게 했는지”에 대한 근거 항상 작성하기
    • 기술 선택과 설계 결정에 대한 ADR(Architecture Decision Records) 작성하기
    • 다른 개발자들의 기술 블로그나 컨퍼런스 발표를 통해 다양한 관점 학습하기

  • 커뮤니티 참여

    • 관심 있는 기술 커뮤니티의 온/오프라인 모임 정기적으로 참석하기
    • 내년에는 작은 규모라도 기술 발표 도전해보기
